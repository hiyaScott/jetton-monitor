## 1. 产品定义

### 1.1 架构定位

```
┌─────────────────────────────────────────────────────────────┐
│                     架构关系图                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────────┐      HTTP GET      ┌───────────────┐ │
│   │  Jetton Monitor │  ───────────────► │  网页仪表盘    │ │
│   │  (桌面客户端)    │  ◄───JSON数据────  │  (数据源)      │ │
│   │                 │                    │  保持现状不动  │ │
│   └─────────────────┘                    └───────────────┘ │
│                                                             │
│   • 桌面应用 = 纯客户端，只负责展示和交互                      │
│   • 网页端 = 数据源，通过 JSON 端点暴露数据                    │
│   • 两者解耦，互不影响                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 核心问题
用户在多个平台与AI助手交互时，无法直观了解助手的实时工作状态：
- 助手是否在线？
- 当前负载如何？
- 消息发送后大概需要多久回复？
- 是否有后台任务在执行？

### 1.2 解决方案
Jetton Monitor 是一个常驻桌面的轻量级监控工具，通过系统托盘和悬浮窗口，实时展示AI助手的认知负载状态。

### 1.3 核心价值主张

```
┌─────────────────────────────────────────────────────────────┐
│  像查看CPU占用一样，实时了解你的AI助手状态                      │
│                                                             │
│  🟢 0-25%   → 立即响应，随时打扰                             │
│  🔵 25-45%  → 轻微负载，30秒内回复                           │
│  🟡 45-65%  → 中等忙碌，简单任务优先                         │
│  🔴 65%+    → 高负载状态，建议等待                           │
└─────────────────────────────────────────────────────────────┘
```

## 2. 用户画像

### 2.1 主要用户：AI助手重度使用者
- 每天与AI助手进行多轮对话
- 需要处理复杂任务（代码、写作、分析）
- 在意响应时间和任务完成度
- 使用多个平台（飞书、Telegram、Discord等）

### 2.2 使用场景

| 场景 | 行为 | 需求 |
|------|------|------|
| **工作前** | 查看托盘图标颜色 | 判断是否可以立即开始新任务 |
| **任务中** | 点击托盘查看详情 | 了解当前排队情况，预估等待时间 |
| **离开后返回** | 查看历史走势 | 了解AI在自己离开期间的工作状态 |
| **多机器人管理** | 切换不同实例 | 管理个人和团队的多个AI助手 |

## 3. 功能规格

### 3.1 MVP功能清单

#### P0 - 核心功能（必须实现）

| 功能 | 描述 | 验收标准 |
|------|------|----------|
| **极简配置** | 首次启动只填URL | 单字段输入，3秒完成配置 |
| **实时监控** | 轮询获取AI状态数据 | 30秒刷新，显示认知评分、任务队列 |
| **系统托盘** | 最小化到托盘，图标显示状态 | 托盘图标颜色随负载变化 |
| **仪表盘窗口** | 主展示界面 | 复用Web版UI，优化桌面交互 |
| **智能默认值** | 所有设置都有合理默认值 | 用户不配置也能正常使用 |

#### P1 - 增强功能（MVP后迭代）

| 功能 | 描述 | 优先级 |
|------|------|--------|
| **视图切换** | 收音机样式 ↔ 纯数字模式 | P1 |
| **多实例管理** | 同时监控多个AI | P1 |
| **桌面通知** | 高负载/离线时提醒 | P1 |
| **全局快捷键** | 快速呼出窗口 | P1 |
| **主题切换** | DeepSpace/Daylight | P1 |
| **数据导出** | 历史数据CSV导出 | P2 |
| **窗口置顶** | 保持窗口在最前 | P2 |

### 4.4 核心数据结构

```typescript
// 监控数据类型
interface MonitorData {
  cognitive_score: number;        // 0-100 认知评分
  active_sessions: number;        // 活跃会话数
  pending_count: number;          // 待处理消息数
  processing_count: number;       // 处理中任务数
  total_tokens: number;           // 总Token数
  total_tokens_formatted: string; // 格式化显示
  estimated_response_seconds: number;
  estimated_response_formatted: string;
  cpu_percent: number;
  memory_percent: number;
  task_queue: TaskItem[];
  history_5m: HistoryPoint[];
  history_15m: HistoryPoint[];
  history_1h: HistoryPoint[];
  last_update: string;
}

interface TaskItem {
  id: string;
  name: string;
  label: string;
  status: 'waiting' | 'processing' | 'idle';
  tokens: number;
  last_role: string;
}

interface HistoryPoint {
  timestamp: string;
  score: number;
}

// 配置类型
interface AppConfig {
  instances: MonitorInstance[];
  settings: AppSettings;
}

interface MonitorInstance {
  id: string;
  name: string;
  dataUrl: string;
  refreshInterval: number;
  enabled: boolean;
}

interface AppSettings {
  autoStart: boolean;
  minimizeToTray: boolean;
  theme: 'deepspace' | 'daylight' | 'system';
  notifications: NotificationSettings;
  globalShortcut: string;
}
```

## 5. 数据源规范

### 5.1 数据获取方式

Jetton Monitor 通过 HTTP GET 请求获取JSON数据，支持任何符合以下格式的数据源：

```
GET https://your-domain.com/status-data.json
```

### 5.2 数据源格式

```json
{
  "cognitive_score": 35,
  "active_sessions": 3,
  "pending_count": 1,
  "processing_count": 1,
  "total_tokens": 12500,
  "total_tokens_formatted": "12.5k",
  "estimated_response_seconds": 45,
  "estimated_response_formatted": "45s",
  "cpu_percent": 25,
  "memory_percent": 40,
  "task_queue": [
    {
      "id": "task_1",
      "name": "SRPG分析",
      "label": "SRPG: 计策系统设计",
      "status": "processing",
      "tokens": 12500,
      "last_role": "assistant"
    }
  ],
  "history_5m": [
    {"timestamp": "08:20", "score": 30},
    {"timestamp": "08:21", "score": 35}
  ],
  "last_update": "2026-03-16T08:23:00+08:00"
}
```

### 5.3 自建数据源

用户可以通过以下方式创建自己的数据源：

1. **GitHub Pages**（当前方式）
   - 定时脚本推送JSON到GitHub
   - 通过CDN访问

2. **本地HTTP服务器**
   - Python: `python -m http.server 8080`
   - Node: `npx serve`

3. **OpenClaw插件**（未来）
   - 内置状态API端点
   - 直接暴露监控数据

### 5.4 极简配置设计原则

为了降低用户使用门槛，Jetton Monitor 遵循**"一个字段即可启动"**的设计原则：

#### 用户只需提供

| 字段 | 是否必填 | 说明 |
|------|----------|------|
| **数据 URL** | ✅ 必填 | 唯一需要用户填写的字段 |
| 实例名称 | ❌ 自动 | 从 URL 域名自动提取 |
| 刷新间隔 | ❌ 默认 | 30秒 |
| 主题 | ❌ 默认 | 跟随系统 |
| 启动行为 | ❌ 默认 | 最小化到托盘 |

#### 配置示例对比

**极简模式（推荐）**
```json
{
  "instances": [
    {
      "dataUrl": "https://your-site.com/data.json"
    }
  ]
}
```

**高级模式（可选）**
```json
{
  "instances": [
    {
      "id": "auto-generated",
      "name": "My Jetton",
      "dataUrl": "https://your-site.com/data.json",
      "refreshInterval": 30,
      "enabled": true
    }
  ],
  "settings": {
    "theme": "system",
    "notifications": { "enabled": true }
  }
}
```

#### 智能默认值

| 配置项 | 默认值 | 理由 |
|--------|--------|------|
| 刷新间隔 | 30秒 | 平衡实时性与资源消耗 |
| 最小化到托盘 | 开启 | 桌面工具的标准行为 |
| 开机自启 | 关闭 | 避免打扰，用户主动选择 |
| 主题 | 跟随系统 | 减少用户决策 |
| 通知 | 开启 | 高负载时提醒用户 |

## 7. 交互规格

### 7.1 窗口行为

| 操作 | 行为 |
|------|------|
| **点击关闭按钮** | 最小化到托盘（可配置为退出） |
| **点击托盘图标** | 显示/隐藏主窗口 |
| **右键托盘图标** | 显示上下文菜单 |
| **启动时** | 默认最小化到托盘（可配置） |

### 7.2 快捷键

| 快捷键 | 功能 |
|--------|------|
| `Ctrl/Cmd + Shift + J` | 显示/隐藏主窗口 |
| `Ctrl/Cmd + R` | 立即刷新数据 |
| `Ctrl/Cmd + ,` | 打开设置 |
| `Esc` | 关闭设置窗口 |

### 7.3 数据刷新

- **自动刷新**: 默认30秒
- **手动刷新**: 点击刷新按钮或快捷键
- **失败重试**: 3次指数退避（1s, 2s, 4s）
- **超时**: 10秒无响应视为失败
